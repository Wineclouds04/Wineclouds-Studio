# Wineclouds Studio 模块施工规范

> 本规范定义 Wineclouds Studio 后续功能模块的目录、命名空间与接入方式。新增或改造模块必须遵循本规范；已有模块逐步向该结构靠拢。

## 1. 目标

- 以业务模块组织源码，不以全局技术层目录组织源码。
- 模块内聚：页面、模型、配置、服务和互操作代码放在同一模块根目录下。
- 模块边界清晰：跨模块通用能力进入 `Shared/`，不能为了复用而把业务代码放回根目录。
- 命名统一：目录、命名空间、XAML `x:Class` 和导航类型保持一致。

## 2. 固定目录

```text
src/WinecloudsStudio/
├── Modules/
│   ├── <Feature>/                 # 一个实际业务模块
│   │   ├── Pages/                 # WinUI 页面及其 code-behind
│   │   ├── Models/                # 仅供该模块使用的视图/领域模型
│   │   ├── Configuration/         # 该模块的本地配置、持久化与选项
│   │   ├── Services/              # 该模块的服务实现
│   │   │   ├── Interface/         # 模块内服务抽象（按需创建）
│   │   │   ├── Implementation/    # 服务实现（按需创建）
│   │   │   └── Interop/           # Win32 等平台互操作（按需创建）
│   │   ├── Views/                 # 独立窗体、控件或视图工厂（按需创建）
│   │   └── Core/                  # 独立算法项目（仅在需要单独项目时使用）
│   ├── Home/                      # 首页模块
│   ├── Navigation/                # 导航与兜底页面
│   └── Reserved/                  # 尚未实现的预留模块
├── Shared/                        # 多个模块共同依赖的无业务能力
│   └── Logging/
├── Assets/                        # 应用级静态资源
├── App.xaml                       # 应用入口
└── MainWindow.xaml                # 应用壳与一级导航
```

### 规则

1. 实际模块使用业务名称，例如 `WindowManager`、`ScreenDetection`；禁止使用 `ModuleA`、`ModuleB` 等临时代号作为新模块名称。
2. 预留功能可临时位于 `Modules/Reserved/<ModuleName>/`，实现时必须迁入其正式业务目录。
3. 不创建新的根级 `Pages/`、`Services/`、`Views/`、`Configuration/` 目录；这类代码必须归属某个模块。
4. `Shared/` 只放跨两个及以上模块复用、且不含业务语义的能力。模块 A 不得直接依赖模块 B 的内部服务。
5. 禁止将模块命名为 `System`、`Collections`、`Threading` 等可能遮蔽 .NET 基础命名空间的名称。

## 3. 命名空间与页面命名

目录与命名空间必须一一对应：

| 代码位置 | 命名空间示例 |
| --- | --- |
| `Modules/WindowManager/Pages/WindowManagerPage.xaml.cs` | `WinecloudsStudio.Modules.WindowManager.Pages` |
| `Modules/WindowManager/Models/ProcessItem.cs` | `WinecloudsStudio.Modules.WindowManager.Models` |
| `Modules/ScreenDetection/Services/ScreenCaptureService.cs` | `WinecloudsStudio.Modules.ScreenDetection.Services` |
| `Shared/Logging/Logger.cs` | `WinecloudsStudio.Shared.Logging` |

页面采用 `<Feature>Page` 名称，例如 `ScreenDetectionPage`。XAML 与 code-behind 必须同步更新：

```xml
<Page x:Class="WinecloudsStudio.Modules.ScreenDetection.Pages.ScreenDetectionPage">
```

```csharp
namespace WinecloudsStudio.Modules.ScreenDetection.Pages;

public sealed partial class ScreenDetectionPage : Page
{
}
```

## 4. 新建模块流程

1. 在 `Modules/<Feature>/` 创建模块根目录；只创建当前功能实际需要的子目录。
2. 新页面放入 `Pages/`，并以业务名称命名；模型、配置和服务放入同一模块的对应子目录。
3. 在 `MainWindow.xaml.cs` 引入页面命名空间，并在 `NavigateTo` 中增加导航键到页面类型的映射。
4. 在 `MainWindow.xaml` 增加对应菜单项，菜单的 `Tag` 与 `NavigateTo` 映射键保持一致。
5. 若模块需要独立算法项目，将项目放在 `Modules/<Feature>/Core/`，同步更新：
   - `WinecloudsStudio.slnx` 的项目路径；
   - 应用项目的 `ProjectReference`；
   - 测试项目的 `ProjectReference`（如存在）；
   - 应用项目的 `Compile Remove`，避免主项目重复编译 `Core/` 中的源码。
6. 更新 `README.md` 的项目结构说明，确保文档与源码一致。

## 5. 依赖边界

- 页面可以依赖本模块的模型、配置和服务。
- 服务可以依赖本模块的接口、互操作层和 `Shared/` 能力。
- `Core/` 项目不得依赖 WinUI 页面或应用壳；它应保持可独立编译。
- 跨模块协作应通过明确的共享抽象或事件边界实现，不直接引用其他模块的 `Implementation/`、`Views/` 或页面。

## 6. 完工核对

完成目录调整或新增模块时，至少检查：

- 所有 XAML `x:Class` 与 code-behind 命名空间、类型名一致。
- `MainWindow` 的导航映射能引用目标页面类型。
- `.slnx`、`ProjectReference` 与 `Compile Remove` 路径同步更新。
- 不存在旧目录或旧命名空间引用。
- 仅提交源码与文档；不提交 `bin/`、`obj/`、`artifacts/`、本地配置或敏感信息。

测试、审计与视觉验收由开发者执行；施工阶段优先保持模块边界、项目引用和编译配置正确。
